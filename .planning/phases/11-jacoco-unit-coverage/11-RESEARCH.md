# Phase 11: JaCoCo Unit Coverage - Research

**Researched:** 2026-01-22
**Domain:** Java code coverage with Maven
**Confidence:** HIGH

## Summary

JaCoCo (Java Code Coverage) is the standard code coverage library for Java/Maven projects. The jacoco-maven-plugin version 0.8.14 (latest stable, October 2024) supports Java 21+ and integrates seamlessly with Maven Surefire for unit test coverage.

The critical technical challenge for this phase is **argLine merging**: the project already uses `-XX:+EnableDynamicAgentLoading` in Surefire's argLine for Mockito compatibility. JaCoCo adds its own agent via the same argLine property. The solution is using Maven's late property evaluation syntax `@{argLine}` which allows JaCoCo to prepend its agent while preserving existing arguments.

**Primary recommendation:** Configure JaCoCo with `prepare-agent` and `report` goals, modify Surefire to use `@{argLine} -XX:+EnableDynamicAgentLoading`, and create a bash script that parses `jacoco.xml` with xmllint.

## Architecture Patterns

### Report Output Structure
```
target/
├── jacoco.exec              # Binary execution data (generated by prepare-agent)
└── site/
    └── jacoco/
        ├── index.html       # HTML report entry point
        ├── jacoco.xml       # XML report for CI/SonarCloud
        ├── jacoco.csv       # CSV report (alternative parsing)
        └── [package dirs]/  # Per-package HTML reports
```

### Pattern 1: Late Property Evaluation for argLine Merging
**What:** Use `@{argLine}` syntax in Surefire to allow JaCoCo to prepend its agent
**When to use:** Always when existing argLine configuration exists
**Example:**
```xml
<!-- Source: https://dev.to/khmarbaise/apache-maven-jacoco-configuration-i71 -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <configuration>
        <argLine>@{argLine} -XX:+EnableDynamicAgentLoading</argLine>
    </configuration>
</plugin>
```

### Pattern 2: Single Execution Block for Goals
**What:** Define both prepare-agent and report goals in one execution
**When to use:** Simple unit test coverage without integration tests
**Example:**
```xml
<!-- Source: https://www.eclemma.org/jacoco/trunk/doc/maven.html -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.14</version>
    <executions>
        <execution>
            <goals>
                <goal>prepare-agent</goal>
                <goal>report</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

### Pattern 3: Separate Executions with Explicit Phases
**What:** Define prepare-agent and report as separate executions with explicit phase bindings
**When to use:** When you need fine-grained control or separate unit/integration coverage
**Example:**
```xml
<!-- Source: https://www.eclemma.org/jacoco/trunk/doc/prepare-agent-mojo.html -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.14</version>
    <executions>
        <execution>
            <id>prepare-agent</id>
            <goals>
                <goal>prepare-agent</goal>
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals>
                <goal>report</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

### Anti-Patterns to Avoid
- **Hardcoding JaCoCo agent in argLine:** Don't manually specify `-javaagent:...jacoco...`. Use prepare-agent goal.
- **Overwriting argLine property:** Don't use `<argLine>-XX:+EnableDynamicAgentLoading</argLine>` without `@{argLine}`. This replaces JaCoCo's agent.
- **Using `${argLine}` instead of `@{argLine}`:** The `${}` syntax evaluates at configuration time (too early). Use `@{}` for late evaluation.
- **forkCount=0 in Surefire:** Prevents JaCoCo agent from running. Always fork tests.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Coverage instrumentation | Bytecode manipulation | jacoco-maven-plugin prepare-agent | Agent injection is complex, version-specific |
| Report generation | Manual parsing of .exec files | jacoco-maven-plugin report goal | Binary format is internal, may change |
| XML parsing in bash | grep/sed patterns | xmllint --xpath | XML structure requires proper parsing |
| Coverage thresholds | Manual checking script | jacoco-maven-plugin check goal | Phase 12 will add this |

**Key insight:** JaCoCo's `.exec` binary format is internal and not documented for external parsing. Always use JaCoCo's own tools to generate reports.

## Common Pitfalls

### Pitfall 1: argLine Overwritten by Surefire
**What goes wrong:** JaCoCo sets `argLine` property, but Surefire's hardcoded `<argLine>` overwrites it
**Why it happens:** Surefire config takes precedence over Maven properties
**How to avoid:** Use `@{argLine}` syntax to include JaCoCo's agent
**Warning signs:** Coverage report shows 0% or "Skipping JaCoCo execution due to missing execution data file"

### Pitfall 2: @{argLine} Undefined Error
**What goes wrong:** Running tests without JaCoCo fails with "Could not find or load main class @{argLine}"
**Why it happens:** `@{argLine}` is not replaced when JaCoCo plugin doesn't run
**How to avoid:** Define empty `<argLine></argLine>` property in `<properties>` section
**Warning signs:** Test failures only when running `mvn test` directly without coverage

### Pitfall 3: Report Phase Binding
**What goes wrong:** Running `mvn test` doesn't generate report
**Why it happens:** Default report phase is `verify`, not `test`
**How to avoid:** Explicitly bind report goal to `test` phase
**Warning signs:** `target/site/jacoco/` directory is empty or missing after `mvn test`

### Pitfall 4: Missing jacoco.exec File
**What goes wrong:** Report generation fails or shows 0% coverage
**Why it happens:** prepare-agent didn't run, or tests ran without agent, or forking disabled
**How to avoid:** Ensure prepare-agent runs before tests, verify forkCount > 0
**Warning signs:** No `target/jacoco.exec` file after test run

### Pitfall 5: Class Mismatch at Runtime
**What goes wrong:** Coverage shows 0% for classes that were definitely tested
**Why it happens:** Runtime class differs from compiled class (mocking frameworks, bytecode enhancement)
**How to avoid:** Usually not an issue with standard Mockito; if using PowerMock or similar, exclude affected classes
**Warning signs:** Classes appear in report but show 0% even with passing tests

## Code Examples

Verified patterns from official sources:

### Complete pom.xml Configuration (Recommended)
```xml
<!-- Source: https://www.eclemma.org/jacoco/trunk/doc/maven.html + argLine from DEV Community article -->
<properties>
    <!-- Empty default allows tests to run without JaCoCo -->
    <argLine></argLine>
</properties>

<build>
    <plugins>
        <!-- JaCoCo Coverage Plugin -->
        <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <version>0.8.14</version>
            <executions>
                <execution>
                    <id>prepare-agent</id>
                    <goals>
                        <goal>prepare-agent</goal>
                    </goals>
                </execution>
                <execution>
                    <id>report</id>
                    <phase>test</phase>
                    <goals>
                        <goal>report</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>

        <!-- Surefire with argLine merging -->
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-surefire-plugin</artifactId>
            <configuration>
                <excludes>
                    <exclude>**/*IT.java</exclude>
                </excludes>
                <!-- @{argLine} = JaCoCo agent, then our custom args -->
                <argLine>@{argLine} -XX:+EnableDynamicAgentLoading</argLine>
            </configuration>
        </plugin>
    </plugins>
</build>
```

### Coverage Script - Parsing jacoco.xml with xmllint
```bash
#!/usr/bin/env bash
# Source: JaCoCo XML DTD at https://github.com/jacoco/jacoco/blob/master/org.jacoco.report/src/org/jacoco/report/xml/report.dtd

set -euo pipefail

REPORT_XML="target/site/jacoco/jacoco.xml"

# Run tests with coverage
mvn test -q

if [[ ! -f "$REPORT_XML" ]]; then
    echo "Error: Coverage report not found at $REPORT_XML"
    exit 1
fi

echo "Coverage Summary"
echo "================"

# Extract counter values using xmllint XPath
# Counter types: INSTRUCTION, BRANCH, LINE, COMPLEXITY, METHOD, CLASS
for type in LINE BRANCH METHOD CLASS; do
    missed=$(xmllint --xpath "string(//report/counter[@type='$type']/@missed)" "$REPORT_XML" 2>/dev/null || echo "0")
    covered=$(xmllint --xpath "string(//report/counter[@type='$type']/@covered)" "$REPORT_XML" 2>/dev/null || echo "0")

    total=$((missed + covered))
    if [[ $total -gt 0 ]]; then
        percent=$((covered * 100 / total))
        printf "%-12s: %3d%% (%d/%d)\n" "$type" "$percent" "$covered" "$total"
    else
        printf "%-12s: N/A (no data)\n" "$type"
    fi
done

echo ""
echo "Full report: target/site/jacoco/index.html"
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| JaCoCo .exec format for SonarQube | XML report format only | SonarQube 8.2 (2020) | Must generate XML, not just .exec |
| `${argLine}` syntax | `@{argLine}` syntax | Surefire 2.17 (2014) | Late evaluation prevents conflicts |
| Manual agent configuration | prepare-agent goal | JaCoCo 0.5.3+ (2012) | No need to specify javaagent path |
| Separate HTML/XML goals | Single report goal with formats | JaCoCo 0.8.0+ | One goal generates all formats |

**Deprecated/outdated:**
- **JaCoCo .exec import in SonarQube:** Removed in SonarQube 8.2+. Must use XML format.
- **Manual javaagent configuration:** Not needed since JaCoCo 0.5.3. Use prepare-agent goal.
- **`${argLine}` for merging:** Still works but `@{argLine}` is preferred for late evaluation.

## Open Questions

Things that couldn't be fully resolved:

1. **Optimal report phase binding**
   - What we know: Default is `verify`, but binding to `test` works
   - What's unclear: Any downsides to `test` phase binding for unit tests?
   - Recommendation: Use `test` phase for faster feedback; user wants `mvn test` to generate report

2. **xmllint availability**
   - What we know: Part of libxml2-utils on Ubuntu/Debian, pre-installed on most systems
   - What's unclear: CI runner availability (GitHub Actions Ubuntu runners have it)
   - Recommendation: Document dependency in script; fallback to awk+CSV if needed
