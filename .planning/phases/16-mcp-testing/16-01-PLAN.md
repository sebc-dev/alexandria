Create `McpSchemaSnapshotGenerator` — a utility class (no Spring context needed) that:

1. Instantiates `McpToolService` with dummy/null dependencies (we only need the class for annotation reflection, not runtime behavior).
2. Uses `MethodToolCallbackProvider.builder().toolObjects(toolService).build()` to extract `ToolCallback[]` from the `@Tool`-annotated methods.
3. For each `ToolCallback`, extracts: tool name, description, and the full `inputSchema` (JSON object with properties, required fields, types).
4. Builds a sorted JSON array (sorted by tool name alphabetically) of tool objects: `{ "name": "...", "description": "...", "inputSchema": {...} }`.
5. Serializes to JSON with sorted keys (`ObjectMapper` with `ORDER_MAP_ENTRIES_BY_KEYS` or `SerializationFeature.ORDER_MAP_ENTRIES_BY_KEYS`), 2-space indent.
6. Provides a `public static String generateSchema()` method returning the JSON string.
7. Provides a `public static void writeSchemaToFile(Path path)` for the Gradle task.

To construct McpToolService with null dependencies for annotation scanning only: use Mockito or manual null stubs. Since this is a unit test utility, use `Mockito.mock()` for all 7 constructor parameters. The tool registration only reads annotations, not runtime method invocations.

Then generate the initial `src/test/resources/mcp/tools-schema.json` by running the generator. Verify it contains exactly 7 tools: `search_docs`, `list_sources`, `add_source`, `remove_source`, `crawl_status`, `recrawl_source`, `index_statistics`.

**Key details:**
- Use `ToolCallback.getToolDefinition()` to get `ToolDefinition` which has `name()`, `description()`, `inputSchema()`.
- The `inputSchema()` returns a String (JSON). Parse it to a Map/JsonNode for sorted serialization.
- The generated JSON structure should be an array of objects sorted by tool name, each containing name, description, and inputSchema (as a parsed JSON object, not a string).
- ObjectMapper configured with `INDENT_OUTPUT` (2-space is Jackson default) and `ORDER_MAP_ENTRIES_BY_KEYS`.

**McpToolSchemaSnapshotTest** — a pure unit test (no `@SpringBootTest`, no Spring context):

1. `toolCountGuard()` — Asserts the generated schema contains exactly 7 tools. This fails early if a tool is added/removed, before the JSON comparison.

2. `schemaMatchesReference()` — Loads `src/test/resources/mcp/tools-schema.json` via classloader, generates current schema via `McpSchemaSnapshotGenerator.generateSchema()`, compares the two JSON strings. Uses Jackson `ObjectMapper.readTree()` for structural JSON comparison (not string equals) so whitespace differences don't matter. On mismatch, produce a readable diff message showing which tools were added/removed/modified. For diff formatting: compare tool lists by name, then for matching names compare description and inputSchema fields. Build a human-readable message like:
   ```
   MCP schema has changed. Run `./gradlew updateMcpSnapshot` to update.

   Added tools: [new_tool]
   Removed tools: [old_tool]
   Modified tools:
     - search_docs: inputSchema changed
       expected: {...}
       actual: {...}
   ```

3. `allSevenToolNamesPresent()` — Explicitly checks the 7 expected tool names are present in the schema. Uses a Set of expected names.

**Testing approach:**
- Use `@SuppressWarnings("NullAway.Init")` since no Mockito extension (pure unit test, generator creates its own mocks).

- Follow AAA structure, camelCase naming per project conventions.
