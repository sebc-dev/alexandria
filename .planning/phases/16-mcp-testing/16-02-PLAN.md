**SQL seed data** at `src/integrationTest/resources/mcp/seed-data.sql`:

Create a self-contained SQL script that inserts test data matching the Flyway schema (V1-V4). The script must:

1. Insert 2-3 sources into the `sources` table:
   - Source 1: Spring Boot docs (INDEXED status, has chunks)
   - Source 2: LangChain4j docs (INDEXED status, has chunks)
   - Source 3: React docs (PENDING status, no chunks) — for testing empty/pending scenarios
   - Use fixed UUIDs (e.g., `'00000000-0000-0000-0000-000000000001'`) for predictable test assertions.

2. Insert 5-10 document_chunks with **real pre-computed BGE embeddings** (384 dimensions):
   - To generate embeddings: use the BGE bge-small-en-v1.5-q model from the application. Write a small utility or manual step note. Alternatively, embed short representative texts and hard-code the resulting float arrays in the SQL.
   - **Pragmatic approach**: Since we need real embeddings for vector search to work, generate them by:
     a. Writing short test chunk texts (e.g., "Spring Boot auto-configuration", "LangChain4j embedding model setup")
     b. Using the ONNX model at test runtime to embed these texts — BUT this means the seed must include placeholder embeddings that are realistic enough for cosine similarity to work.
     c. **Simpler alternative**: Use random but normalized 384-dim vectors. The search will return results based on cosine similarity, which with random vectors will give low but non-zero scores. For testing that search returns SOMETHING, this is sufficient. The round-trip test verifies the MCP transport, not search quality.
   - Include metadata JSONB with `source_url`, `section_path`, `content_type`, `source_name` fields.
   - Link chunks to sources via `source_id` FK.
   - Include both prose and code content types for filter testing.

3. SQL structure:
   ```sql
   -- Clean slate (idempotent)
   DELETE FROM document_chunks WHERE source_id IN ('00000000-...001', '00000000-...002');
   DELETE FROM sources WHERE id IN ('00000000-...001', '00000000-...002', '00000000-...003');

   -- Sources
   INSERT INTO sources (id, url, name, status, chunk_count, last_crawled_at) VALUES (...);

   -- Chunks with embeddings
   INSERT INTO document_chunks (embedding_id, embedding, text, metadata, source_id) VALUES (...);
   ```

**Important for embeddings**: Use the pgvector literal format `'[0.1, 0.2, ..., 0.384th]'::vector(384)`. For realistic-enough vectors, generate 384 random floats between -0.1 and 0.1, then normalize to unit length. Include 5-8 chunks across the 2 indexed sources. Each chunk needs distinct text content that matches its metadata.

Create `McpRoundTripIT` — a `@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)` integration test that:

**Test infrastructure setup:**
- Extends or reuses `BaseIntegrationTest` pattern (Testcontainers pgvector with `@ServiceConnection`)
- Uses `@ActiveProfiles("web")` to activate the MCP SSE transport
- Uses `@Sql("/mcp/seed-data.sql")` to load test data before each test (or `@BeforeAll` with manual SQL execution)
- Injects `@LocalServerPort` to get the random port
- Creates `McpSyncClient` in `@BeforeAll` or `@BeforeEach`:
  ```java
  var transport = HttpClientSseClientTransport.builder("http://localhost:" + port).build();
  var client = McpClient.sync(transport).build();
  client.initialize();
  ```
- Closes client in `@AfterAll` or `@AfterEach`

**Important SSE endpoint path:** Spring AI MCP WebMVC SSE endpoint is typically at `/sse`. So the transport URL should be `http://localhost:{port}` (the SDK appends `/sse` itself based on the SSE endpoint discovery). Verify by checking Spring AI MCP server auto-configuration — the SSE endpoint is at the root or `/sse`. Use `HttpClientSseClientTransport.builder("http://localhost:" + port)` which connects to the root and follows the SSE endpoint.

**Test helpers:**
- `callTool(String name, Map<String, Object> args)` — wraps `client.callTool(new CallToolRequest(name, args))` and extracts text from `TextContent` in the result.
- `callTool(String name)` — no-args variant using empty map.
- `assertNoError(CallToolResult result)` — asserts `isError()` is null or false.
- `extractText(CallToolResult result)` — gets the first TextContent text.

**Tests for each tool (~20-25 total):**

**search_docs** (3-4 tests):
- `searchDocsHappyPathReturnsResults` — search with a query matching seeded chunks, verify response contains text and source URLs
- `searchDocsWithFiltersBySource` — search with source filter, verify results are from that source only
- `searchDocsWithEmptyQueryReturnsError` — call with empty/null query, verify error message (not JSON-RPC error)
- `searchDocsWithNoMatchesReturnsNotFound` — search for non-existent topic, verify "No results" message

**list_sources** (2 tests):
- `listSourcesReturnsAllSources` — verify response contains all seeded source names and statuses
- `listSourcesIncludesChunkCounts` — verify chunk count information is present

**add_source** (2-3 tests):
- `addSourceCreatesNewSourceAndStartsCrawl` — add a source, verify response confirms creation with ID
- `addSourceWithEmptyUrlReturnsError` — verify error message for blank URL
- `addSourceWithScopeParamsAcceptsThem` — add source with allow/block patterns, verify response

**remove_source** (2-3 tests):
- `removeSourceDeletesExistingSource` — remove a seeded source, verify confirmation with chunk count
- `removeSourceWithInvalidUuidReturnsError` — verify error for malformed UUID
- `removeSourceWithNonExistentIdReturnsNotFound` — verify error for unknown UUID

**crawl_status** (2 tests):
- `crawlStatusForIndexedSourceShowsSummary` — check status of seeded source, verify shows INDEXED status and chunk info
- `crawlStatusWithInvalidUuidReturnsError` — verify error for malformed UUID

**recrawl_source** (2-3 tests):
- `recrawlSourceTriggersIncrementalRecrawl` — trigger recrawl on seeded source, verify response confirms incremental start
- `recrawlSourceWithNonExistentIdReturnsNotFound` — verify error for unknown UUID
- `recrawlSourceWithInvalidUuidReturnsError` — verify error for malformed UUID

**index_statistics** (2 tests):
- `indexStatisticsReturnsGlobalStats` — verify response contains chunk count, source count, storage size
- `indexStatisticsContainsEmbeddingDimensions` — verify "384" appears in response

**Testing patterns:**
- Each test follows AAA (Arrange-Act-Assert) with single-line Act
- Use `assertThat(text).contains(...)` for response content validation — MCP tool methods return plain text strings, not structured JSON
- For tools that modify state (add_source, remove_source, recrawl_source): order matters. Tests that create sources should use unique URLs to avoid conflicts. Tests that remove sources should work on specifically seeded data. Consider using `@Sql` with `executionPhase = BEFORE_TEST_METHOD` and `config = @SqlConfig(transactionMode = ISOLATED)` for clean state.
- For `recrawl_source`: the `dispatchCrawl` will start a virtual thread that attempts to crawl (and will fail since Crawl4AI is not running). The test only verifies the synchronous response ("Recrawl started..."), not the crawl outcome.
- For `add_source`: similarly, the async crawl will fail, but the synchronous response is what we test.
- camelCase test method names per project convention.
- `@SuppressWarnings("NullAway.Init")` at class level for test fields.

**Crawl4AI sidecar concern:** The Crawl4AI sidecar is NOT running in tests. Tools that trigger crawls (add_source, recrawl_source) dispatch async virtual threads that will fail silently. The round-trip test verifies the synchronous MCP response, not the async crawl outcome. This is by design — we test the MCP transport contract, not the crawl pipeline.

**Important concurrency note:** Since add_source and recrawl_source dispatch background virtual threads that modify source status, add a small delay or use `@DirtiesContext` for tests that modify state. Alternatively, use `@Order` or separate test data to avoid interference. Preferred: use distinct fixed UUIDs for seed data and test-created data, so tests don't interfere.
